# 1. 产品基本信息

## 1.1 产品定位

## 结论

**箩筐体育 1.0 的核心定位是：面向足球爱好者，提供“高频查比赛 + 高效率看数据 + 结构化看资讯”的足球数据与资讯平台。**

### 定位说明

* **产品核心价值**
  以“比赛”为主入口，以“赛事/球队/球员”为数据沉淀对象，以“资讯”为内容补充，满足用户从赛前、赛中、赛后到日常关注的完整信息获取需求。
* **产品侧重点**
  1.0 版本采取 **数据优先、资讯辅助** 的策略，而非纯资讯社区路线。
* **差异化方向**
  与泛体育信息平台相比，箩筐体育更强调：

  * 比赛、赛事、球队、球员四层数据关系的清晰串联
  * 资讯与数据对象的强关联
  * PC 端与移动端一致的信息架构
  * 低学习成本、高信息密度、强检索效率

### 这里基于假设

* 假设 1.0 阶段暂不以社区互动、直播、视频为主增长点，而是先建立“可查、可信、可看懂”的基础能力。
* 假设首期目标市场以中文用户为主，因此资讯表达、本土联赛展示、中文搜索体验需优先优化。

## 1.2 核心场景

### 核心场景 1：赛前查比赛

用户在比赛开始前查询：

* 今日/明日比赛安排
* 联赛赛程
* 对阵双方基本信息
* 球队近况
* 伤停/阵容趋势（1.0 可先不做深度伤停，仅保留基础阵容信息位）
* 关联资讯

### 核心场景 2：赛后看数据

用户在比赛结束后查看：

* 比分与结果
* 比赛关键统计
* 事件时间线
* 球员表现
* 联赛排名变化
* 与比赛相关的资讯解读

### 核心场景 3：日常跟踪球队/球员/赛事

用户日常关注：

* 某球队赛程、战绩、阵容
* 某球员赛季数据、能力维度、荣誉
* 某联赛积分榜、射手榜、赛程
* 某类资讯列表

### 核心场景 4：碎片化刷资讯

用户在通勤、休息等碎片时间浏览：

* 热门足球资讯
* 联赛/球队/球员相关资讯
* 与比赛强关联的新闻

## 1.3 目标客户

## 核心用户画像

### 用户类型 A：重度足球数据用户

* 年龄：20-40 岁为主
* 特征：经常看比赛、熟悉联赛与球队，强依赖赛程、积分榜、球员数据
* 核心诉求：查得快、数据准、关系清楚、维度足够

### 用户类型 B：大众足球资讯用户

* 年龄：18-35 岁为主
* 特征：不一定天天看比赛，但会关注热点球队、热门球员和大赛
* 核心诉求：能快速看懂比赛结果、球队近况、重点新闻

### 用户类型 C：内容消费与轻研究型用户

* 年龄：22-45 岁
* 特征：关注某一联赛或某几个球队，偏好长期跟踪
* 核心诉求：赛事、球队、球员、资讯之间跳转顺畅，沉浸式浏览效率高

## 1.4 商业模式

## 1.0 版本建议

**1.0 版本以验证用户价值与留存为主，商业化不作为首要 KPI，但需预留接口。**

### 商业模式规划

* **广告变现**

  * 首页信息流广告位
  * 资讯列表页原生广告位
  * 赛事/球队详情页品牌曝光位
* **会员订阅（P1/P2）**

  * 更深度的数据维度
  * 自定义关注与提醒
  * 高级筛选与对比功能
* **B 端数据服务（P2）**

  * 数据 API
  * 内容分发合作
  * 媒体/自媒体嵌入组件

### 建议

1.0 不建议过早堆叠重商业化模块，优先保证数据可信度、页面可用性和日活留存。

---

## 1.5 版本目标

## 1.0 版本目标

建立箩筐体育的最小可用产品（MVP），完成“比赛—赛事—球队—球员—资讯”五大信息对象的闭环。

### 核心目标

* 用户能高效完成比赛信息查询
* 用户能从任一信息对象进入其他关联对象
* 用户能在 3 次点击内抵达核心详情页
* 用户能在数据页面中自然消费关联资讯
* 用户能在资讯页面中自然回跳关联比赛/球队/球员/赛事

### 关键指标建议

* 首页比赛卡点击率
* 比赛详情页到球队/球员/赛事页跳转率
* 资讯详情页阅读完成率
* 次日留存 / 7 日留存
* 页面平均停留时长
* 搜索转化率（P1 可上）

# 3. 产品风格设计

# 3.1 整体布局设计

## PC 端布局原则

* 顶部：全站主导航 + 搜索入口 + 主题切换
* 左中区域：主内容区，承载列表、详情、图表、正文
* 右侧区域：辅助信息区，承接热门资讯、相关球队/球员、榜单入口
* 底部：版权、合作、免责声明、下载入口

## 移动端布局原则

* 顶部：页面标题 + 返回/搜索/筛选
* 中部：核心内容流，按“信息优先级”垂直组织
* 底部：主 Tab 导航
* 弹层：筛选、榜单切换、赛季选择等使用底部弹层实现

## 3.3 UI 规范

### 字体层级

* H1：28/32，页面主标题
* H2：22/28，模块标题
* H3：18/24，卡片标题
* Body：14/20，正文/说明
* Caption：12/16，辅助信息

### 间距规范

* 基础单位：8px
* 卡片内边距：16px / 20px
* 模块间距：24px
* 移动端左右安全边距：16px

### 圆角与阴影

* 卡片圆角：12px
* 弹层圆角：16px
* 阴影：低对比度轻阴影，避免过度拟物

### 组件规范

* Tab：支持吸顶与横向滚动
* 筛选器：单选/多选统一使用 Chip 样式
* 列表卡片：统一信息顺序与状态标识
* 表格：优先支持固定表头、横向滚动
* 图标：线性图标为主，局部配合实心状态点

---

## 3.4 UX 准则

1. **比赛优先原则**
   用户最常访问的是比赛与关联对象，主路径必须最短。

2. **对象串联原则**
   比赛、赛事、球队、球员、资讯之间必须能互相跳转。

3. **低记忆负担原则**
   筛选、赛季切换、榜单切换等状态需可见、可恢复。

4. **信息层次明确原则**
   先给结果，再给解释；先给核心数据，再给扩展数据。

5. **跨端一致原则**
   PC 与移动端信息架构一致，仅布局方式调整。

6. **加载可感知原则**
   关键页面必须有骨架屏、空状态、异常状态、重试机制。

## 4.2 1.0 范围内典型旅程

首页比赛中心 → 选择日期/联赛 → 点击比赛卡 → 进入比赛详情页 → 查看数据 → 跳转球队/球员详情页

# 5.1 首页（比赛中心页）

## 页面目标

承接全站最高频访问需求，帮助用户快速找到目标比赛。

## 页面定位

全站主入口、流量中心、比赛分发中心。

## 页面布局

### PC

* 顶部：LOGO、主导航、搜索、主题切换
* 左侧/主区域：

  * 日期切换
  * 联赛筛选
  * 比赛列表
* 右侧：

  * 热门资讯
  * 热门赛事入口
  * 热门球队/球员入口
* 底部：分页/加载更多、版权信息

### 移动端

* 顶部：标题、日期切换、筛选按钮
* 中部：比赛列表卡片流
* 底部：Tab 导航

## 展示元素

### 顶部导航

* 首页
* 比赛
* 赛事
* 资讯
* 搜索入口（P1）
* 明暗主题切换

### 筛选区

* 日期：昨天 / 今天 / 明天 / 日历选择
* 联赛筛选：热门联赛 + 更多
* 状态筛选：全部 / 未开赛 / 进行中 / 已结束
* 排序：按时间 / 按关注热度（P1）

### 比赛卡片字段

* 联赛名称
* 比赛时间
* 状态（未开赛/进行中/完场）
* 主队名称 + Logo
* 客队名称 + Logo
* 比分
* 半场比分
* 关键标记：红牌/点球/加时/取消
* 快捷入口：数据、阵容、资讯

## 功能点摘要

* 比赛列表浏览
* 筛选与切换
* 比赛详情跳转
* 联赛详情跳转
* 热门资讯引流

## 交互逻辑

* 日期切换后，列表整体刷新
* 联赛筛选支持多选
* 状态筛选与日期筛选并行生效
* 点击比赛卡任意主体区域进入比赛详情
* 点击联赛名称进入赛事详情页
* 右侧资讯点击进入资讯详情页

## 状态设计

* 加载中：骨架屏
* 无数据：展示“暂无比赛”
* 接口异常：重试按钮
* 超长列表：PC 分页 / 移动端加载更多

# 12. 最终结论

## 结论

**箩筐体育 1.0 应定义为“以比赛为入口、以数据为骨架、以资讯为补充”的足球数据资讯产品。**
在首版中，不追求大而全，而应优先把以下三件事做好：

1. **把比赛相关查询体验做到顺畅**
2. **把赛事、球队、球员三类对象页的数据结构打通**
3. **把资讯和数据对象形成强关联闭环**

这样做的价值在于：

* 既能满足高频的看比赛、查数据需求
* 又能承接日常碎片化资讯消费
* 同时为后续关注、提醒、直播、会员等功能留出自然扩展路径
